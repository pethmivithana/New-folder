LOGGING & DEBUGGING IMPROVEMENTS — ML MODEL INFERENCE
================================================================================

PURPOSE:
  Expose silent inference failures in ML models by adding detailed error logging
  and debugging output. Users will now see exactly where/why ML predictions fail
  instead of getting silent hardcoded fallback values (e.g., always 32% quality).

================================================================================
KEY FIXES IMPLEMENTED
================================================================================

ISSUE #1: Silent Errors (Root Cause Analysis)
  Status: FIXED
  Before: Exceptions caught but only printed simple message, then returned hardcoded value
  After: Full traceback + context logged to stderr for debugging

ISSUE #2: TabNet Data Types (float64 → float32 conversion)
  Status: FIXED
  Before: build_quality_features() returned float64 arrays (PyTorch default)
          TabNet expects float32 and would silently fail
  After: 
    - Explicit dtype=np.float32 in np.array() call
    - .astype(np.float32) after StandardScaler transform (for productivity)
    - Validation assertions before return

ISSUE #3: Array Shape Mismatches (1D → 2D)
  Status: FIXED  
  Before: Potential 1D arrays passed to models expecting 2D
  After:
    - All feature builders return 2D arrays: (1, n_features)
    - Validation assertions: assert X.shape == (1, 6) for quality, etc.

ISSUE #4: Missing Visibility Into Model Inference
  Status: FIXED
  Before: No logging of what models are actually running in production
  After:
    - Each model logs input arrays before inference
    - Each model logs output values after inference
    - Errors include model type, shapes, dtypes, and full stack trace
